# Orchestration Evidence Standards (OpenClaw Delegation)

## Goal

Define what counts as proof that openclaw delegated work to the orchestration system and that final PR changes were driven by that flow.

This is a gate for PR reviews, incident audits, and any future automation claims.

## Scope

- Applies to work claimed as "executed by openclaw" in PRs, docs, and status updates.
- Focuses on end-to-end evidence: task delegation, orchestration execution, and artifact traceability.
- Does not replace code review; it defines what evidence must exist before we accept a “fully delegated” claim.

## Delegation Contract

For a change to be marked "delegated end-to-end," openclaw must:

1. Create at least one `openclaw_chat_async` or equivalent task enqueue event.
2. Route that task into `ai_orch`/agent executor.
3. Produce an execution artifact that maps back to the task (patch, run output, or commit hash).
4. Avoid untracked, out-of-band manual edits to task files unless explicitly justified.

If manual edits are required, the PR must use the status `delegated with manual follow-up`.

## Evidence Checklist (Required in PR)

### A) Orchestration Queue Evidence

- Task enqueue record from `openclaw_chat_async` (request id + timestamps).
- `openclaw` or gateway log entry with task creation and payload summary.
- If asynchronous routing was used, capture task id correlation (`task_id`, `run_id`, or similar).

### B) Executor Evidence

- `ai_orch` execution logs showing:
  - executor selected (model/CLI),
  - session/worktree identity,
  - lifecycle state (`started`, `completed`, `failed`, `killed`).
- Command exit status and timeout behavior for each run.
- If the run was split across workers, include each worker id and combined output.

### C) Artifact Evidence

- Diff artifact from the orchestration step:
  - explicit patch content produced by the worker, **or**
  - explicit file-level edit summary emitted by the worker, **or**
  - commit hash generated by worker PR branch.
- Files changed in PR must be explainable from that artifact.

### D) Handoff Evidence

- Final handoff message that maps task(s) → PR or branch.
- Any blocking/manual fallback reason and exactly what was done outside the delegated run.

## Hard Failure Conditions

The following invalidate a claim of “openclaw did it end-to-end”:

- No enqueue record exists.
- Executor log is missing for claimed scope.
- Task-to-diff mapping is ambiguous.
- Logs explicitly state fallback language such as:
  - “completed directly”
  - “not applied by executor”
  - “manual edit” without context and justification.

## PR Review Outcome Mapping

- `delegated`: all requirements in A–D are present and mapped.
- `delegated with manual follow-up`: all A/B present, C/D include explicit manual intervention rationale and changed file mapping.
- `not delegated`: one or more A/B artifacts missing or unverifiable.

## Required PR Attachments

- `orchestration_task_ids` (list)
- `executor_run_ids` (list)
- `artifacts` (patch links, log excerpts, run output file paths)
- `manual_edits` (`none` or `yes` + reason)
- `evidence_timestamp_window` (UTC range)
